# Build-n-Break

Give a task to an AI executor, then run an adversarial code reviewer against the output. You review disputes, not the full diff.

## Usage

```
/build-n-break [task description]
```

$ARGUMENTS

## Workflow

### Phase 1: Build

1. If no task description was provided, ask for one
2. Record the current HEAD commit as the baseline:
   ```bash
   BASELINE=$(git rev-parse HEAD)
   ```
3. Execute the task. Implement the requested changes — write code, create files, modify existing code. Work through the full implementation as you normally would. Commit your work as you go.
4. Record the final commit:
   ```bash
   FINAL=$(git rev-parse HEAD)
   ```

### Phase 2: Break

5. Spawn the `adversarial-code-reviewer` agent with the commit range `$BASELINE..$FINAL`:

   Prompt the agent with:
   ```
   Review the changes in {BASELINE}..{FINAL}.

   These changes were just generated by an AI agent for this task:
   {task description}

   Hunt for reckless completion patterns, fabricated assumptions, scope violations,
   and anything that looks plausible but might be wrong. Verify your suspicions
   before flagging them.
   ```

### Phase 3: Report

6. Present the results as a dispute report:

```markdown
## Build-n-Break Report

### What was built
{Brief summary of the implementation}

### Commits
{BASELINE}..{FINAL} ({N} commits)

### Adversarial Review

{Paste the adversarial code reviewer's findings}

### Disputes for Human Review
{List only the items marked Red Flag or Suspicion — these are the things
that need your eyes. Verified OK items are noted but not highlighted.}

### What to check
{Prioritized list of specific things the human should verify,
based on the adversarial findings. Be concrete:
"Check that X actually works by doing Y" not "verify correctness."}
```

## Guidelines

- **Don't pull punches in the break phase.** The whole point is to find problems before they ship. An adversarial review that finds nothing is fine — but it should have looked hard.
- **The build phase is normal implementation.** Don't sandbag the build to make the break easier. Implement the task properly, then let the adversary do its job.
- **Disputes are the value.** The human's time should go to adjudicating the disputes the adversary raised, not scanning the entire diff. If the adversary found nothing, say so and the human can decide if that's sufficient.
- **Keep the report actionable.** "What to check" should be specific enough that the human can verify each item in under 5 minutes.
